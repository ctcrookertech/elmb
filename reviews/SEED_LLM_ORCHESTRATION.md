# ELMB Seed — LLM Orchestration Review

**Expert**: Dr. Anya Petrov, LLM Orchestration & Agent Loop Design (18 years experience)
**Scope**: seed/machine.go, seed/cmd_elmb.go, seed/learn.go, seed/model.go, seed/build.go, core/out.go, infer/cmd_infer.go
**Date**: 2026-02-28

## Executive Summary

The ELMB state machine implements a four-mode adaptive loop (Enact, Learn, Model, Build) with an elegant arise/relax flow between modes and frame-based context accumulation. However, the system has critical gaps in its LLM integration layer: it relies on directive-only output from models without system prompts or format enforcement, it has an incomplete adaptive loop where Build cannot re-trigger Enact, and it exposes API keys in process arguments. These issues collectively mean the system will behave unpredictably under real workloads and cannot complete its core adaptive scenario without manual intervention.

## Findings

### 1. Prompt Robustness — Directive Parsing Under Real LLM Output

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 1.1 | **Silent loss of non-directive LLM output.** All three directive parsers (`parseLearnDirectives`, `parseModelDirectives`, `parseBuildSteps`) skip lines that do not match their expected prefix patterns. When the LLM produces preamble ("Here are my observations:"), markdown formatting ("## Plan"), numbered lists ("1. Run the build"), or wraps directives in code blocks (`` ```\nSTEP: ...\n``` ``), all of that content is silently dropped. The `core.Line(core.Learn, "skipping unrecognized line: "+line)` trace exists but only in Learn and Model — Build's `parseBuildSteps` silently ignores non-STEP lines with no trace at all. | **High** | (a) Add a system prompt to every infer call that constrains the model to directive-only output (see Finding 9). (b) Add a pre-processing pass that strips common LLM formatting artifacts: markdown fences, leading bullet markers (`- `, `* `, numbered `1. `), and leading whitespace indentation before attempting directive parsing. (c) Track the ratio of parsed-to-skipped lines; if more than 50% of non-empty lines are skipped, log a warning and consider re-prompting. |
| 1.2 | **No retry or re-prompt on zero valid directives.** If the LLM produces a response where every line is skipped, Learn will arise with the original content unchanged, Model will arise with the original item (the `!hasPlan` path), and Build will add the raw content to the frame. None of these outcomes indicate failure — they silently degrade. | **Medium** | Add an explicit check: if the infer call returns non-empty text but zero valid directives were parsed, treat this as a format error. Log it prominently and optionally retry with a more constrained prompt (e.g., prepending "You MUST output ONLY directives. No explanations."). |

### 2. Incomplete Adaptive Loop — Build Cannot Re-Trigger Enact

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 2.1 | **The full adaptive cycle is broken.** The intended scenario is: Enact runs command X and it fails -> Learn extracts the failure -> Model creates a plan including "build a new tool" -> Build creates the tool -> the system re-enacts with the new tool. Tracing through the code: `processEnact` runs a command, captures output, pushes to frame, arises to Learn. `processLearn` extracts facts, arises to Model. `processModel` creates PLAN/STEP directives, arises to Build. `processBuild` iterates steps and spawns child `elmb` instances for actionable steps. **But Build never pushes anything to ModeEnact.** It only calls `m.framePush()` and `m.spawnSync()`. The spawned children are independent processes with their own state machines — their results are captured as strings and added to the parent's frame, but never fed back as Enact items. The parent's Enact stack is empty after the initial drain, and nothing re-populates it. | **Critical** | After Build completes its steps, it should evaluate whether the original goal has been achieved. If a tool was created, Build should push a new `Item` onto `m.Stacks[ModeEnact]` with the appropriate `Command` and `Args` to run the newly created tool. This closes the loop. Example: after a step that creates a tool `foo`, push `Item{Command: "foo", Args: []string{...}, Source: core.Build}` onto the Enact stack. |
| 2.2 | **Build's child spawns use ModeModel as limit.** In `processBuild`, child `elmb` processes are spawned with `Limit: ModeModel` (line 71 of build.go). This means the child can Enact, Learn, and Model, but not Build. If the child's Model phase produces a plan requiring Build, it will be added to the frame instead (via `arise` hitting the limit). This prevents recursive tool creation. | **Medium** | Consider whether `Limit: ModeBuild` is more appropriate for some step types, or allow the step text to specify the limit. At minimum, document why ModeModel is chosen as the ceiling. |

### 3. Model-to-Learn Relaxation — Semantic Mismatch

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 3.1 | **INVESTIGATE relaxes a question to Learn, but Learn expects command output.** When `processModel` encounters an `INVESTIGATE` directive, it calls `m.relax(ModeModel, Item{Content: d.Text, Source: core.Model})` where `d.Text` is an investigation question like "What version of Go is installed?". This item is pushed onto the Learn stack. But `processLearn` constructs prompts that say "You are analyzing output from a command execution" and "New content to analyze: [the question text]". The LLM will be asked to extract facts from a question, which is semantically wrong — a question is not output, and the extract prompt will likely produce confused or hallucinated "facts." | **High** | When Model relaxes with INVESTIGATE, the item should go to Enact with a command that can answer the question (e.g., by inferring what command to run), not to Learn. Alternatively, Learn should detect `Source: core.Model` and switch to an investigation-specific prompt that treats the content as a question to answer rather than output to analyze. |
| 3.2 | **INVESTIGATE causes early return, discarding remaining directives.** In `processModel` (line 82), when an INVESTIGATE directive is encountered, the function immediately returns. If the LLM produced `PLAN`, `STEP`, `STEP`, `INVESTIGATE`, `STEP`, the final STEP is lost. The PLAN and first two STEPs are already pushed to the frame, but the arise never happens, so Build never receives them. | **Medium** | Collect all directives first, then decide the action. If INVESTIGATE is present alongside PLAN/STEP, either prioritize investigation (and save the partial plan for later) or execute the plan and investigate in parallel. |

### 4. Build Step Classification — Over-Conservative Actionability

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 4.1 | **`looksActionable()` uses strict prefix matching against 10 verbs.** The function checks `strings.HasPrefix(lower, v)` for verbs like "run ", "execute ", "call ". But LLMs phrase steps in many ways: "First, run the test suite", "Use the infer tool to query...", "Check whether the service is running", "Try executing...", "Set up the environment by running...". None of these match the prefix list. The function also misses imperative forms common in plans: "start", "stop", "delete", "remove", "write", "generate", "test", "download", "fetch", "send". | **High** | (a) Expand the verb list significantly, or switch to substring matching (`strings.Contains`) for action verbs anywhere in the step text. (b) Better yet, have the Model phase explicitly mark steps as actionable using a structured directive, e.g., `STEP-RUN: run tests` vs `STEP-INFO: ensure tests pass`. This moves classification to the LLM, which understands the intent. (c) As a fallback, add a "verb anywhere" heuristic: if any action verb appears as a word boundary in the step, treat it as actionable. |
| 4.2 | **Non-actionable steps are silently added to frame without execution.** When a step is classified as informational, it is pushed to the frame as a `LevelStep` element with no further processing. There is no feedback to the Model that these steps were not executed. In a plan like "STEP: Verify the output format / STEP: Run the test suite", the verification step is skipped but the frame grows as if it was completed. | **Medium** | At minimum, prefix informational frame entries with "skipped: " or "informational: " to distinguish them from executed results. Consider logging a warning when more than half of steps are classified as informational, as this likely indicates a classification problem. |

### 5. Frame Pollution — Monotonic Growth of Context

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 5.1 | **Every mode unconditionally pushes to the default frame.** `processEnact` pushes command output (line 183 of machine.go). `processLearn` pushes extracted facts (line 116 of learn.go). `processModel` pushes PLAN and STEP entries (lines 86, 91 of model.go). `processBuild` pushes step results and informational steps (lines 65, 83 of build.go). In a multi-cycle run, the frame accumulates raw command output, extracted facts, plan titles, step descriptions, and build results — all at the same level of the prompt context. | **High** | (a) Use the `FrameLevel` enum that already exists but is never filtered on. Future prompts should include only relevant levels (e.g., Model should see LevelProc facts but not LevelStep build artifacts from prior cycles). (b) Apply compaction more aggressively — the threshold of 10 is reasonable only if entries are short. Consider a token-count-based threshold instead. (c) Implement frame scoping: when a new cycle begins (Enact processes a new seed), archive or namespace the previous cycle's frame entries. |
| 5.2 | **Frame content is included in every prompt with no relevance filtering.** `frameText("")` dumps the entire default frame into every Learn, Model, and compaction prompt. As the frame grows, prompts become dominated by stale context, increasing token cost and reducing the LLM's ability to focus on the current item. | **Medium** | Add a recency window or relevance filter. At minimum, limit the frame context to the last N entries (e.g., 20) or the last K tokens. Entries beyond the window can be summarized into a single "historical context" line. |

### 6. Compaction Quality — LLM-Dependent Frame Surgery

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 6.1 | **Compaction relies on the LLM producing exact `= <low>-<high>: <replacement>` syntax.** This is a precise format with integer ranges that must be valid indices into the current frame. If the LLM produces `= 0-5: summary` when the frame only has 4 entries, or produces `= 2-1: text` (inverted range), data is silently dropped or corrupted. The bounds check (line 196 of learn.go) catches out-of-range and inverted ranges, but overlapping ranges applied in reverse order can still corrupt: e.g., `= 0-5: A` then `= 3-8: B` applied in reverse would first replace 3-8, shifting indices, then replace 0-5 which now includes the replacement from the second operation. | **High** | (a) Validate that ranges do not overlap before applying. Sort by `Low` and check for intersections. (b) Consider a simpler compaction format: have the LLM output a complete replacement list rather than surgical range directives. E.g., "Output the compacted entries, one per line, prefixed with `KEEP: `". This eliminates index arithmetic entirely. (c) Add a checksum or entry count verification after compaction to detect silent corruption. |
| 6.2 | **Reverse-order application assumes independence.** The comment "Apply in reverse order so indices stay valid" (line 191) is correct only if ranges do not overlap and are sorted. There is no sort and no overlap check. The directives arrive in whatever order the LLM produced them. | **Medium** | Sort directives by `Low` descending before applying, and reject any directive whose range overlaps with a previously applied range. |

### 7. API Key Exposure via Command-Line Arguments

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 7.1 | **API key passed as `os.Args[1]` to the infer binary.** In `inferDirect` (machine.go line 265): `m.runCommandWithInput("infer", []string{m.APIKey, "-"}, prompt)`. In `processLearn` (learn.go lines 97-98): `Args: []string{m.APIKey, "-"}`. In `processBuild` (build.go line 73): `Args: []string{m.APIKey, "-"}`. The API key appears in the process argument list, visible via `ps aux`, `/proc/*/cmdline`, and system monitoring tools. On shared systems, any user can read another user's process arguments. | **High** | Pass the API key via environment variable (`ELMB_API_KEY` is already read in `cmd_elmb.go`). Have the infer tool read from the environment instead of accepting it as a CLI argument. This is a one-line change in infer's `main()` and eliminates the exposure entirely. Alternatively, pass the key via stdin as a header line before the prompt, or via a temporary file with restricted permissions. |
| 7.2 | **Child elmb spawns do not propagate the API key.** In `spawnSync` and `spawnAny`, the child `elmb` is invoked with `--plain --limit <mode> <command> <args>` but no API key argument. The child resolves its own API key via `resolveAPIKey()` which reads `ELMB_API_KEY` or the `anthropic.key` file. This works only if the environment variable is inherited or the key file exists in the child's working directory. If neither condition holds, child processes silently degrade to passthrough mode ("no api key, passthrough"). | **Medium** | Explicitly pass the API key to children via environment variable in the `exec.Cmd.Env` field, ensuring consistent behavior regardless of working directory or environment inheritance. |

### 8. Infer Output Parsing — Round-Trip Reliability

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 8.1 | **LLM response text containing `[exoutput]` or `[  output]` will corrupt parsing.** `parseOutputBlock` (machine.go line 429) scans for `[  output]` to start capture and `[exoutput]` to stop. If the LLM's response text contains the literal string `[exoutput]` — which is possible if the LLM is asked about the ELMB system itself, or if it echoes parts of its prompt — the parser will prematurely terminate capture, truncating the response. Similarly, a literal `[  output]` in the response would not cause issues (it just continues capturing) but is still a protocol confusion risk. | **High** | (a) Use a unique sentinel that cannot appear in LLM output, such as a UUID generated per invocation. (b) Alternatively, have infer output a length-prefixed or base64-encoded block instead of delimiter-based framing. (c) As a minimal fix, escape the sentinel strings in the LLM's output before wrapping them in the block protocol. |
| 8.2 | **ANSI stripping is fragile.** The ANSI stripper in `parseOutputBlock` (lines 432-443) only handles `\033[...m` sequences (SGR). Other ANSI sequences like cursor movement (`\033[H`), erase (`\033[2J`), or OSC sequences (`\033]...`) will not be stripped and will appear as garbage in the parsed output. While infer only uses SGR via `core.Tag`, other sibling commands run via `runCommand` might emit different ANSI sequences. | **Low** | Use a proper ANSI stripping regex: `\033\[[0-9;]*[a-zA-Z]` or a dedicated function that handles all CSI sequences. |
| 8.3 | **`BlockEnd` emits a leading newline.** In `core/out.go` line 55: `fmt.Printf("\n%s\n", Tag(ExOutput))`. This means the captured block always ends with an empty line. `parseOutputBlock` captures everything between `[  output]` and `[exoutput]`, so this trailing empty line becomes part of the parsed content. For directive parsing this is benign (empty lines are skipped), but for content that is used verbatim (e.g., frame entries), it introduces trailing whitespace. | **Low** | Trim trailing whitespace from `parseOutputBlock` output before returning. |

### 9. No System Prompt — Unconstrained Model Behavior

| # | Finding | Severity | Recommendation |
|---|---------|----------|----------------|
| 9.1 | **All infer calls send prompts as user messages with no system prompt.** In `infer/cmd_infer.go` line 39: `"messages": []map[string]string{{"role": "user", "content": input}}`. There is no `system` field in the API request body. The model receives directive-format instructions ("Output only directives, no other text") as part of the user message, which is the weakest position for format compliance. Models are significantly more likely to follow output format constraints when they appear in the system prompt. | **Critical** | Add a `system` field to the API request. Move the format instructions (directive syntax, "output only directives") to the system prompt. Keep the contextual data (frame contents, content to analyze) in the user message. This is the single highest-leverage change for improving reliability across the entire system. Example: `"system": "You are a component in an automated pipeline. You MUST output ONLY directives in the specified format. Do not include explanations, preamble, markdown formatting, or any other text."` |
| 9.2 | **No structured output or tool-use API features.** The Anthropic API supports tool use (function calling), which would allow defining the exact output schema. Instead of hoping the LLM produces `STEP: ...` lines, the system could define a tool with a `steps` array parameter, guaranteeing structured output. | **Medium** | For critical directive types (Learn extract, Model plan, compaction), consider using tool-use mode to get structured JSON output. This eliminates all parsing fragility at the cost of slightly more complex API integration. |

## Overall Assessment

**Grade**: C-

The ELMB architecture demonstrates a thoughtful theoretical design — the four-mode loop with arise/relax transitions is an elegant model for adaptive autonomous agents. The frame-based context accumulation and the hierarchical depth limits show awareness of the recursive execution risks inherent in agent loops. The convention-based build system and output protocol are clean and well-structured.

However, the implementation has fundamental gaps that prevent it from functioning reliably as an autonomous agent. The two most severe issues are: (1) the absence of system prompts in LLM calls, which makes every directive parser unreliable in practice — LLMs will routinely produce preamble, markdown formatting, and non-directive text that is silently discarded; and (2) the broken adaptive loop where Build cannot re-trigger Enact, meaning the system's core value proposition (enact, learn from failure, model a new approach, build a tool, re-enact) cannot actually complete without human intervention. The API key exposure in process arguments is a significant security concern for any deployment beyond local development. The compaction mechanism, while clever in concept, uses LLM-generated index arithmetic that is prone to silent data corruption. Taken together, these issues mean the system will appear to run but will silently lose information, fail to close its adaptive loop, and produce inconsistent results depending on the LLM's formatting choices on any given invocation.

## Recommendations Summary

1. **[Critical] Add system prompts to all LLM calls.** Move format instructions to the `system` field in the Anthropic API request. This is the single highest-impact change — it improves directive compliance across Learn, Model, Build, and compaction without touching any other code. (Findings 9.1, 1.1)

2. **[Critical] Close the adaptive loop.** After Build creates a tool or completes actionable steps, push a new Enact item onto the stack to re-execute with the new capability. Without this, the system cannot achieve its stated design goal. (Finding 2.1)

3. **[High] Stop passing API keys as command-line arguments.** Switch infer to read from `ELMB_API_KEY` environment variable. Explicitly set the env var in child process `exec.Cmd.Env` to ensure propagation. (Findings 7.1, 7.2)

4. **[High] Fix Model-to-Learn relaxation semantics.** When INVESTIGATE relaxes to Learn, the Learn prompts must recognize that the content is a question to investigate, not command output to analyze. Either route investigations through Enact or add source-aware prompt selection in Learn. (Finding 3.1)

5. **[High] Harden directive parsing against LLM formatting.** Add a pre-processing pass to strip markdown fences, bullet markers, and numbered list prefixes. Track parse success rates and warn on high skip ratios. (Findings 1.1, 1.2)

6. **[High] Fix `parseOutputBlock` sentinel collision.** The `[exoutput]` delimiter can appear in LLM output. Use a unique per-invocation sentinel, length-prefixed framing, or escape the delimiters in content. (Finding 8.1)

7. **[High] Improve Build step classification.** Expand `looksActionable()` substantially, or better yet, have Model emit structured `STEP-RUN:` vs `STEP-INFO:` directives so classification happens at the semantic level. (Finding 4.1)

8. **[High] Make compaction safer.** Replace index-range-based compaction with a simpler "output the compacted list" format that eliminates index arithmetic. If ranges are kept, validate for overlaps and sort before applying. (Findings 6.1, 6.2)

9. **[Medium] Address frame pollution.** Filter frame entries by `FrameLevel` when constructing prompts for different modes. Add a token budget or recency window to prevent unbounded prompt growth. (Findings 5.1, 5.2)

10. **[Medium] Consider structured output via tool use.** The Anthropic API's tool-use feature would eliminate all directive parsing issues by returning structured JSON. Evaluate for the highest-value call sites (Model planning, Learn extraction). (Finding 9.2)
